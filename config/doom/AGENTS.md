# Personal Configuration Agent Guidelines

This document contains guidelines for agents (human or AI) modifying the
personal Emacs configuration.

## Modular Architecture by Topic

The configuration follows a modular design where settings, hooks, and package
setup are divided strictly by topic. Instead of a single massive configuration
file, each major area (e.g., AI integrations, specific application settings, or
Org-mode components like roam, agenda, or capture templates) has its own
dedicated `.org` file. This keeps the configuration organized, readable, and
easier to maintain.

## Ongoing Migration Status

The migration from the monolithic `config-personal.org` to modular topic-based
include files is currently **ongoing**. Some configuration sections are still
defined inside the main `config-personal.org` file, while others have been
successfully moved into separate `config-personal-*.org` files and are
referenced via `#+INCLUDE:` directives. When editing or adding configuration,
prefer modularity and move settings out of `config-personal.org` into a separate
file if they form a logical, cohesive group.

## Crucial Rules for Modifying Configuration

### 1. Never Edit Elisp Files Manually

You **MUST NOT** manually edit any `config-personal*.el`, `init-personal.el`, or
`packages-personal.el` files. These files are automatically generated by
Org-mode's tangling engine. Any manual changes to these files will be silently
overwritten during the next tangling process.

### 2. Always Edit the Respective Org File

To make a change, you must locate and edit the specific `.org` file that
contains the literate programming source block for the configuration you wish to
modify.

### 3. Tangle From the Top-Level Entry Point

Because of the recursive `#+INCLUDE:` directive structure used to assemble the
modular files, you **MUST** tangle from the top-level
**[config-personal.org](file:///google/src/cloud/cashweaver/emacs/google3/experimental/users/cashweaver/dotfiles/files/doom/config-personal.org)**.
This ensures that the `cashpw/org-tangle-with-includes` custom advice properly
expands all descendant includes and generates complete, synchronized Emacs Lisp
outputs (`config-personal.el`, `init-personal.el`, `packages-personal.el`).
Tangling a sub-file directly will result in incomplete files and broken
configurations.

## Debugging

This section documents common failure modes and root causes encountered when configuring Doom Emacs in the Google environment.

### Startup/Loading Errors with `google-disable-legacy-loading`

*   **Symptom**: Emacs fails to start or throws errors like `Cannot open load file` or `Symbol's function definition is void` for Google-internal or third-party libraries (e.g., `vc-fig`, `go-guru`, `bazel`).
*   **Root Cause**: Setting `google-disable-legacy-loading` to `t` disables the legacy loading mechanism, which prevents Google-internal libraries from being automatically added to the `load-path`. If these packages are not declared as packages in Doom (which uses `straight.el` for package management), Emacs will fail to locate and load them, halting the startup process or breaking features.
*   **Recommended Resolution**: Explicitly declare the missing dependent library as a local package in the appropriate packages block (tangled from the `.org` file), pointing `:recipe` to its local repository in `/usr/share/emacs/site-lisp/emacs-google-config/`.
    *   Example for `vc-fig`:

        ```elisp
        (package! vc-fig
          :recipe (:local-repo "/usr/share/emacs/site-lisp/emacs-google-config/devtools/editors/emacs/vc"))
        ```
    *   Example for `go-guru`:

        ```elisp
        (package! go-guru
          :recipe (:local-repo "/usr/share/emacs/site-lisp/emacs-google-config/third_party/elisp/go"))
        ```

*   **Tracking Bug**: `b/22574484`. Review this tracking bug and its child bugs
    for the most up-to-date status on legacy loading deprecation and transition
    guides.

### Org Agenda and Buffer-Safety (org-element-at-point warnings)

*   **Symptom**: Opening the Org Agenda view (`SPC d d`) triggers warnings like:
    `■ Warning (org-element): ‘org-element-at-point’ cannot be used in non-Org
    buffer #<buffer *Org Agenda*> (org-agenda-mode)`
*   **Root Cause**: Org Agenda is a composite view that aggregates items from
    diverse buffers (including non-Org sources like diary files). Any custom
    functions evaluated on agenda lines (e.g., inside `org-agenda-prefix-format`
    or custom sorting strategies `org-agenda-cmp-user-defined`) run inside the
    buffer context of the respective agenda item. If called in a non-Org buffer
    (e.g., a diary entry), standard metadata accessors like `org-entry-get`,
    `org-entry-properties`, or `org-get-tags` will implicitly call the low-level
    `org-element-at-point` parser, triggering the warning.
*   **Shadowing Override Trap**: In split environments (e.g., where
    `config-work.el` is loaded after `config-personal.el`), duplicate
    definitions in the work configuration will override personal fixes.
*   **Actionable Guidelines**:

    1.  **Double-check Overrides**: Run a global search for `defun
        [function-name]` across the entire repository. Ensure the safety fix is
        applied to the work configuration (`config-work.org`) if it overrides
        the personal one.
    2.  **Check Buffer Mode**: Wrap all custom agenda formatting helpers and
        property accessors in a `(derived-mode-p 'org-mode)` check. Return safe
        fallback spacers or defaults immediately if evaluated in a non-Org
        buffer.

        *   *Safe Icon Helper Example*:

            ```elisp
            (defun cashpw/org-agenda-icon ()
              (if (not (derived-mode-p 'org-mode))
                  "  "
                (let ((properties (org-entry-properties (point))))
                  ...)))
            ```

### Headless Interactive Verification (Daemon + `emacsclient` via Virtual Terminals)

Employ this technique when:

1.  **Debugging AST or UI Formatting**: Changes made to active buffers, custom
    sorting strategies, prefix formatters, or hook functions need empirical
    runtime verification.
2.  **Diagnosing Headless Startup Errors**: Emacs crashes or warnings occur
    during startup that cannot be easily captured in batch mode (`emacs -batch`)
    due to complex package-loading state requirements.
3.  **Verifying Client-Server Integrations**: You need to verify that
    configuration changes load correctly into a persistent daemon without
    disrupting the user's active editing environment.

#### Verification Playbook:

*   **Initialize an isolated virtual terminal session** using `tmx2` or `tmux`
    to prevent background Emacs display processes from blocking or hanging the
    primary shell terminal.
*   **Define a distinct socket name** for the test session (e.g., `jetski-test`)
    to guarantee complete process isolation and prevent accidental connections
    to the developer's active editing daemon.
*   **Launch the headless Emacs server** inside the virtual terminal using
    `emacs --daemon=<socket-name>` to satisfy Emacs' startup terminal capability
    checks (`TERM=screen` / `TERM=xterm-256color`).
*   **Verify the server is listening** by checking for the socket file under the
    user's runtime directory or by running `emacsclient -socket-name
    <socket-name> --eval 't'`.
* **Trigger the dynamic expression or command under investigation** headlessly using `emacsclient -socket-name <socket-name> --eval '<expression>'` (e.g., executing a formatting helper, reloading a module, or compiling a buffer).
* **Audit the server's execution log** by programmatically dumping and filtering the `*Messages*` buffer using `emacsclient -socket-name <socket-name> --eval '(with-current-buffer "*Messages*" (buffer-string))'`.
* **Scan the log dump** specifically for low-level AST warnings (like `org-element-at-point`) or missing macro definitions (like `use-package!`).
* **Terminate the test server cleanly** upon completion by running `emacsclient -socket-name <socket-name> --eval '(kill-emacs)'` to prevent orphan processes.
* **Close the virtual terminal session** using `tmux kill-session` or exiting the `tmx2` pane to clean up system environment resources.
